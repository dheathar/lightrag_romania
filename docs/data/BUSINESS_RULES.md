# Business Rules and Invariants

## Embedding Consistency Invariant

**Rule:** The embedding model and dimension must not change after the first document has been indexed.

**Why:** All stored vectors must share the same dimensionality. Changing the embedding model invalidates all existing vectors. Query-time embeddings must match index-time embeddings for similarity search to produce meaningful results.

**Enforcement:** Not enforced programmatically at the API level. If violated, queries may silently return wrong results or raise a dimension mismatch error depending on the backend.

**Remediation:** Clear vector storage and re-index all documents. The LLM response cache (`kv_store_llm_response_cache.json`) can be preserved to avoid re-paying LLM extraction costs.

**Source:** `lightrag/llm/openai.py` — `openai_embed()`, CLAUDE.md "Common Issues" section.

---

## Entity Type Taxonomy Invariant

**Rule:** Changing `ENTITY_TYPES` requires re-indexing all documents.

**Why:** Entity types are part of the extraction prompt. Existing graph nodes have types extracted under the old taxonomy. New types from a changed taxonomy do not retroactively update existing nodes.

**Enforcement:** Configuration can be changed at runtime via `PUT /config` or env var, but the graph is not rebuilt automatically.

**Remediation:** After changing entity types, either:
1. Clear the graph and KV storage, then re-ingest all documents
2. Accept a mixed-taxonomy graph (acceptable for additive type additions)

**Source:** `lightrag/constants.py:DEFAULT_ENTITY_TYPES`, `env.example:ENTITY_TYPES`

## `<SEP>` Separator Invariant

**Rule:** The `GRAPH_FIELD_SEP = "<SEP>"` separator used in `source_ids`, `description`, `file_path`, and relation `id` fields cannot be changed after data has been inserted.

**Why:** All graph reads split on `<SEP>`. Changing it would corrupt all existing multi-value fields.

**Source:** `lightrag/constants.py:GRAPH_FIELD_SEP`

---

## Concurrency Safety Rule

**Rule:** Relation lock keys must always be generated by sorting the entity pair alphabetically.

**Why:** If `A→B` and `B→A` generate different lock keys, concurrent insertion of both directions creates two locks for the same edge and causes deadlock or duplicate edges.

```python
# Always do this:
sorted_key_parts = sorted([src_entity, tgt_entity])
lock_key = f"{sorted_key_parts[0]}-{sorted_key_parts[1]}"
```

**Source:** `lightrag/kg/shared_storage.py`, CLAUDE.md "Critical Implementation Patterns"

---

## Async Generator Lock Rule

**Rule:** Locks must not be held across `yield` points in async generators.

**Why:** Holding a lock while yielding control to other coroutines blocks the entire lock, preventing concurrent access. The correct pattern is to take a snapshot under the lock and then yield outside it.

**Source:** CLAUDE.md "Async Generator Lock Management (CRITICAL)"

## LLM Context Size Rule

**Rule:** `OLLAMA_LLM_NUM_CTX` must be greater than `MAX_TOTAL_TOKENS + 2000`.

**Why:** The LLM context window must be large enough to hold the assembled context plus system prompt overhead. Ollama defaults to 8K context, which is insufficient for the default `MAX_TOTAL_TOKENS=30000`.

**Recommended minimum:** `OLLAMA_LLM_NUM_CTX=32768`

**Source:** `env.example:OLLAMA_LLM_NUM_CTX`

---

## Token Budget Constraint

**Rule:** `MAX_ENTITY_TOKENS + MAX_RELATION_TOKENS` must be less than `MAX_TOTAL_TOKENS`.

**Why:** The chunk token allocation is dynamically computed as:
```
chunk_tokens = MAX_TOTAL_TOKENS - actual_entity_tokens - actual_relation_tokens
```
If entity + relation tokens consume the entire budget, no chunk context is included.

**Default budget split:**
- `MAX_ENTITY_TOKENS = 6000`
- `MAX_RELATION_TOKENS = 8000`
- `MAX_TOTAL_TOKENS = 30000`
- Available for chunks: up to `30000 - 6000 - 8000 = 16000` tokens

**Source:** `lightrag/constants.py`, `env.example` comments.

---

## Document Deduplication Rule

**Rule:** Documents with identical content (same MD5 hash) are marked `DUPLICATED` and not re-indexed.

**Why:** Prevents duplicate entities and inflated weights from repeated ingestion of the same content.

**Source:** `lightrag/api/routers/document_routes.py` — pipeline_enqueue_file()

---

## Source IDs Limit Rule

**Rule:** Each entity and relation stores at most `MAX_SOURCE_IDS_PER_ENTITY` (default 300) chunk IDs.

**Why:** Unbounded growth of `source_ids` fields would make graph nodes extremely large and slow to serialize/deserialize.

**Eviction policy:** Controlled by `SOURCE_IDS_LIMIT_METHOD`:
- `FIFO`: Remove oldest IDs first (default — more merge activity but fresher provenance)
- `KEEP`: Ignore new IDs once limit is reached (less merge overhead)

**Source:** `lightrag/constants.py`, `lightrag/utils.py:apply_source_ids_limit()`

---

## LLM Summary Trigger Rule

**Rule:** When an entity's or relation's description accumulates `FORCE_LLM_SUMMARY_ON_MERGE` (default 8) segments, an LLM call is triggered to summarize them into a single description.

**Why:** Without summarization, descriptions grow indefinitely with each new mention, increasing storage and prompt costs.

**Trade-off:** Triggers extra LLM calls during ingestion. Increasing the threshold reduces cost but allows longer descriptions.

**Source:** `lightrag/constants.py:DEFAULT_FORCE_LLM_SUMMARY_ON_MERGE`, `lightrag/operate.py:merge_nodes_and_edges()`

---

## Parallel Insert Safety Rule

**Rule:** `MAX_PARALLEL_INSERT` must be at most `MAX_ASYNC / 3` (recommended).

**Why:** Each parallel document insert spawns multiple concurrent LLM calls. Setting `MAX_PARALLEL_INSERT` too high relative to `MAX_ASYNC` causes all LLM semaphore slots to be consumed, starving query requests.

**Default:** `MAX_PARALLEL_INSERT=2`, `MAX_ASYNC=4`.

**Source:** `env.example` comments.

---

## Chunking Size Stability Rule

**Rule:** Chunk size and overlap should not be changed without re-indexing.

**Why:** Chunk boundaries determine which entities and relations are extracted together. Changing chunk size changes which co-occurrences are captured, leading to inconsistent entity descriptions across old and new chunks.

**Recommended range:** `CHUNK_SIZE=500–1500`, `CHUNK_OVERLAP_SIZE=50–200`.

**Source:** `env.example` comments.

---

## Workspace Name Constraints

**Rule:** Workspace names may only contain characters `a-z`, `A-Z`, `0-9`, and `_`.

**Why:** Workspace names are used as prefixes in database identifiers (table names, collection names, Neo4j labels). Special characters would break SQL, MongoDB, and Neo4j naming rules.

**Source:** `env.example:WORKSPACE` comment, `lightrag/kg/shared_storage.py`
